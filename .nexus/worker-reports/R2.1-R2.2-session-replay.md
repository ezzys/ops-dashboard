## R2.1 — Event Store + Session Replay

### R2.1.1 Event Log Schema
- **DB:** `data/events.db` (separate from sessions.db)
- **Mode:** WAL + SYNCHRONOUS=NORMAL for write throughput
- **Schema:** `events` table with all required columns + 5 indexes
  - `idx_events_trace`, `idx_events_surface`, `idx_events_type`, `idx_events_agent`, `idx_events_ts`
- **Methods:** `init()`, `writeEvent()`, `getEventsByTrace()`, `getRecentEvents()`, `queryEvents()`, `getStats()`, `close()`

### R2.1.2 Session Replay Routes

| Route | Description |
|-------|-------------|
| `GET /api/sessions` | List sessions with optional filters (agent_id, model, status, limit, offset) |
| `GET /api/sessions/:id` | Session details enriched with surface event counts |
| `GET /api/sessions/:id/events` | All events for session, chronological |
| `GET /api/sessions/:id/export?format=json\|html` | Download session data |

- Sessions without a `sessions.db` record are synthesised from trace events (graceful degradation)
- HTML export includes inline-styled table, no external dependencies

### R2.1.3 Event Ingestion

| Endpoint | Description |
|----------|-------------|
| `POST /api/events` | HTTP ingestion — single event or `{ events: [...] }` batch (max 500) |
| `WS /ws?room=events` | Real-time ingestion — `type:event`, `type:batch`, or raw object |

- Validates with `event-types.validateEvent()` before writing
- Broadcasts accepted events to `logs` WebSocket room for live subscribers
- Returns `201` for single, `200/207` for batch (207 on partial failure)

### R2.1.4 Data Retention
- `startPruning()` runs `setInterval` every 6 hours
- Deletes events older than `retention.days` (default 30) from `nexus-config.json`
- Timer uses `.unref()` so it doesn't block process exit
- Pruning called in `start()` after event store init

## R2.2 — Three-Surface Event Types

`src/services/event-types.js` exports:

### Cognitive surface
`reasoning_step`, `tool_selected`, `confidence_score`, `context_built`

### Operational surface
`tool_call`, `tool_result`, `function_invoked`, `error`, `retry`

### Contextual surface
`http_request`, `db_query`, `cache_hit`, `cache_miss`, `file_read`, `file_write`

### Validation
- `validateEvent(event)` — full schema check, returns `{ ok, error }`
- `isValidSurface(s)`, `isValidEventType(t)`, `isValidSurfaceEventPair(surface, type)`
- `EVENT_TYPE_TO_SURFACE` — reverse lookup map

## R2.3 — Log Query Interface

| Route | Description |
|-------|-------------|
| `GET /api/logs/query` | Filtered + paginated event query with all filter params |
| `GET /api/logs/search?q=<term>` | Full-text substring search across `data` JSON |
| `GET /api/logs/recent?limit=N` | Last N events across all surfaces |
| `GET /api/logs/stats` | Total + per-surface counts |
| `GET /api/logs/surfaces` | Returns taxonomy (no auth required) |

- Query supports: `surface`, `event_type`, `agent_id`, `trace_id`, `from`, `to`, `search`, `limit`, `offset`
- LIKE-based substring match on `data` column (SQLite built-in, no FTS5 dependency)
- Live stream: all ingested events are broadcast to existing `logs` WebSocket room

## Architecture Notes

- **Separation of concerns:** `event-store.js` (persistence) ↔ `event-ingest.js` (pipeline) ↔ `event-types.js` (schema)
- **Append-only discipline:** No UPDATE or non-pruning DELETE paths exposed via any route
- **Graceful degradation:** Sessions list works even if `sessions.db` is absent (synthesises from event traces); event store fails open (empty results, no crash)
- **WS broadcast is injected:** `setBroadcast(fn)` called after WebSocket setup — no circular require

## Blockers / Notes

- `GET /api/sessions` returns empty `sessions: []` when `sessions.db` is not found — this is expected (same as Phase 1 sessiondb behaviour)
- Old `dashboard.js` (PID 688) occupies port 18790; new `src/server.js` tested on :18793 with identical result set
